# 0002 — 设备身份：持久短 ID 唯一键 + 显示名映射

**Status**: accepted（2026-07-15）

每台运行 VaultOne 的设备在首次启动时生成一个**持久 12 位十六进制短 ID（48 bit）**作为 `deviceId`，用作 Artifact 目录名（`data/<deviceId>/`）与多设备合并的唯一键；看板显示名在 config 中以 `deviceId → 友好名` 单独映射，不参与唯一性。

## 为什么

多设备同步的命门是设备身份的稳定与全局唯一：身份重复或漂移会让数据归属错乱。UUID v4（36 字符 / 122 bit）对个人几台设备严重过剩；12 位十六进制（48 bit）按生日悖论对数千设备碰撞可忽略，且目录名简洁、文件系统 / git / URL 友好。目录名直接用唯一键（而非用户起名）杜绝重名冲突；显示名单独成层，丑 ID 不影响看板可读性。

## 考虑过的替代方案

- **完整 UUID**：唯一性最强，但 36 字符对个人设备场景冗长、目录名难看。
- **机器特征哈希**（主机名 / MAC / 用户名）：自动免存储，但特征变化（换网卡、改主机名）即漂移，且 MAC 进哈希有隐私顾虑。
- **用户手动起名**：直观，但无法防重名——两台同名即同步冲突。

## 后果

- `deviceId` 持久存储于本地 config；重装系统 / 清缓存会丢失，导致历史 Artifact 成为孤儿。**待定**：是否支持「认领旧设备身份」（用旧目录名绑定新安装）。
- 物理迁移（换机器）时旧数据归属**待定**。
- 生成时做一次冲突检测（查已有 deviceId）兜底碰撞，概率极低。
